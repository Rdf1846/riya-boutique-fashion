What is the difference between int and Integer?
Ans1 : The difference between Int and Integer is that - int is a primitive datatype, while INTEGER is the corresponding wrapper class for the int datatype. Basically int values needs some utility methods like .toValue(), .toString(), etc. Now we need a place which can contains these methods, but we can't store them in int so the INTEGER class provides these methods. Also, we can store limited size value in int datatype, while INTEGER is a class, so the range is very very high.

Q4. Why are Entity class fields kept private?
Ans : We usually keep the entity class fields as private to achieve the abstraction. Means if entity fields are not private, then anybody can access their values using .get() or getters. Now, this can be a risk of sensitive information leak like email, password, etc. Instead we create DTO's in which we only create the fields which can be share by other classes easily. We want to invoke the Entity class by the Service Class only, which we basically implements the whole business logic.



Q5. What is Constructor Injection? Why is it preferred over Field Injection?
Ans: So constructor injection is a type of Dependency injection. Basically to inject the dependency of any class into a leader class, we pass the object refrence of that class into the leader class. Now, this object is not created by us, it will automatically creates and managed by the Spring container - Beans. Controll of the objects is under the spring, rather than us. This property is also known as IOC(Inversion of control).
Example of C.I.:
Public class UserService {
private final UserRepository userRepo;

public UserService(UserRepository userRepo)
{
this.userRepo=userRepo;
}

Here, we are the injecting the object of the UserRepo class into the UserService class, using the constructor of the UserService class. This is known as Construction Injection.

Now, here the object reference we created is final, so it can't be modified. But if we use field injection using the @Autowired annotation, then it can't be final. That's why we mostly preferred the CI over other dependency injections.

Q9. Why do we create Repository Interface? Who provides its implementation?
Ans : We create repository interface to talk to the database. Basically Whenever any request body come to our application, we need some mechanism to insert the request into the database. We are not going to write the SQL statement manually everytime. Instead we will make use of the method of this interface. The beauty of this is that the implementation of these methods is provided by the spring during the runtime. With the help of JPA hibernate it generates the sql query automatically to insert the records into the database. 
We are not providing the implemnentation, spring itself providing the implementation at runtime.

Section D - DTO / Mapper
Q15. Why don't we return Entity directly from Controller? Why create DTO?
Ans : Entity represents the whole user data from the database. It contains sensitive information as well like email, password, dob, etc. Even table schemas. So we usually not expose the entity class to the controller class. Instead we create a new class which is having only the fields which we want to share. This new class is known as DTO (Data Transfer Objects). If we pass entity directly anybody can access the fields of the entity class, so map entity to dto, and passes the dto's to controller class.


Q16.Why did we create two DTOs? SignupRequestDto, UserDetailsDto.
Ans : Here, when user signup to our application then we need all information to save in our database even including passwords, email, etc. So, we created a signup dto > duplicate of our entity class. All fields are same as in our ENTITY class. Its a RequestDto type.
Now, when anybody searches the records from the database, then we should be careful that sensitive informations shouldn't be leaked. So here we created userDetailsDto from which we removed sensitive informations like passwords, etc. And, we pass this object as a Response from our application.
